# Contributing to goBrain

goBrain is deliberately modest in scope: ingest AI conversations, embed them,
expose them through `search_brain`. Contributions that stay inside that
mission — and that preserve the **local-first**, **markdown-first**, **swappable**
design — are welcome.

## Before you start

- Open an issue describing the change you want to make, especially if it
  touches the data model or adds a new external dependency.
- Small bug fixes and documentation improvements don't need a pre-discussion.

## Development setup

```bash
# Clone
git clone https://github.com/<your-fork>/goBrain.git
cd goBrain

# The ingester and MCP server are independent Python packages, both using uv.
# Work on whichever you're changing.
cd ingester      # or mcp-server
cp .env.example .env  # fill in your local paths + DSN
uv sync
uv run pytest    # run tests (CI runs these on PR)
```

### Running the full stack locally

See the Quickstart in the root [README.md](README.md). For development you can
collapse all three tiers onto one machine:

- Postgres via Docker compose (`compose/postgres/`)
- Ollama locally
- Ingester in the foreground: `uv run brain-ingester` (skip the LaunchAgent)
- MCP server registered with Claude Code

## Code style

- Python: [ruff](https://github.com/astral-sh/ruff) is configured in
  `pyproject.toml`. Run `uv run ruff check .` and `uv run ruff format .`
  before committing.
- Keep imports tidy — no unused imports, local imports inside functions where
  circular import risk exists.
- Prefer small, focused modules. Don't add abstractions before the second use.

## Commit messages

We keep commit messages descriptive and factual. A good one:

- States the *why* in the first line (under 70 chars)
- Explains what changed and what the alternatives were in the body
- Mentions the trade-off if one exists

No Co-Authored-By footers generated by AI tools. Sign off however you like.

## What's in and out of scope

**In scope:**
- Additional ingestion sources (more AI tool export formats, new agent
  platforms, etc.)
- Better retrieval (reranker improvements, graceful-degradation heuristics,
  new MCP tools)
- Ops tooling (the dashboard, backfill, monitoring)
- Docs, examples, setup scripts for more environments

**Out of scope:**
- Re-architecting to require cloud services (the project is explicitly local-first)
- Replacing markdown as the canonical vault format
- Adding UI frameworks or build systems that require compilation steps for the
  web dashboard (stays plain HTML/JS + CDN assets)

## Testing

There are no tests yet — the initial build was scoped for a single user's
reference deployment. If you're contributing a feature, please add tests for it;
this is how we bootstrap the test suite. `pytest` is wired up via `uv sync`'s
dev deps.
