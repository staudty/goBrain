# Architecture

## Goals

1. **Unify AI context.** Claude Code, Claude Desktop, Claude.ai web, and Grok all draw on the same memory.
2. **Respect the model's token budget.** A small local LLM does the librarian work (embed, summarize, rerank); the primary model only sees tight, curated context.
3. **Local-first and private.** No vault content leaves your network without an explicit call you authorize.
4. **Minimally manual.** Ingestion is automatic for everything we watch directly; for the rest (Grok, Claude.ai exports) a single "drop file in `_inbox/`" is the entire user step.

## Data model

### The vault (markdown)

The human-facing source of truth. Lives at a path you configure (e.g. `<vault-root>/Brain/`), synced to clients via Synology Drive, Syncthing, Dropbox, rsync — whatever works for you.

```
Brain/
├── _inbox/                 # drop zone for manual exports (Grok, Claude.ai, PDFs, etc.)
│   └── _processed/         # ingester moves files here after successful ingestion
├── sessions/               # one note per ingested conversation
│   ├── claude-code/
│   │   └── YYYY-MM-DD_<project>_<short-id>.md
│   ├── claude-desktop/
│   ├── claude-ai/
│   └── grok/
├── .claude-code-sources/   # cross-machine CC JSONLs land here via your sync layer
├── notes/                  # your own notes, written directly in Obsidian
└── templates/              # Obsidian templates for consistent formatting
```

Each session note starts with YAML frontmatter used by both humans and the ingester:

```yaml
---
source: claude-code      # claude-code | openclaw | claude-desktop | claude-ai | grok | inbox
source_id: <stable id>   # for dedup on re-ingestion
project: my-project      # optional context tag
started_at: 2026-04-18T14:03:12-04:00
ended_at: 2026-04-18T15:41:08-04:00
turn_count: 47
tool_calls: 12
model: claude-opus-4-7
tags: [postgres, migrations]
summary: >
  2-sentence TL;DR generated by the local summarizer.
---
```

Body is the conversation, cleaned and condensed. Tool calls rendered in fenced blocks. Long outputs truncated with `<!-- truncated: N lines -->` markers.

### The database (Postgres + pgvector)

The machine-queryable index.

```sql
-- compose/postgres/init.sql drives this; condensed view here:

documents (
  id uuid primary key,
  source text not null,              -- matches frontmatter 'source'
  source_id text not null,           -- stable id for dedup
  vault_path text not null unique,   -- 'sessions/claude-code/2026-04-18_...md'
  started_at timestamptz,
  ended_at timestamptz,
  project text,
  model text,
  turn_count int,
  tool_call_count int,
  summary text,                      -- short summarizer-written tldr
  tags text[],
  raw_hash text,                     -- content hash for change detection
  ingested_at timestamptz default now()
);

chunks (
  id uuid primary key,
  document_id uuid references documents on delete cascade,
  chunk_index int,
  content text not null,
  embedding vector(768)              -- nomic-embed-text dimension
);
```

Indexes:
- `chunks (embedding)` via HNSW for fast ANN search
- `documents (source, started_at desc)` for recency queries
- GIN index on `documents.tags`

### Nothing is thrown away

Raw JSONL logs from Claude Code, raw export ZIPs from Grok/Claude.ai — all preserved under `Brain/_raw/<source>/<date>/` or kept in place at the source (`~/.claude/projects/`). The vault markdown is derived; if you change the summarization logic, you can re-derive.

## Components

### Ollama (always-on host, port 11434)

Hosts the small models. One model loaded at a time (`OLLAMA_MAX_LOADED_MODELS=1`), 60-minute idle unload (tuned up from the 10m default so heavy inbox backfills don't thrash the cache).

| Model | Size on disk | Active RAM | Role |
|---|---|---|---|
| `gemma4:e2b` | ~7 GB | ~4 GB | Classification, rerank, short responses |
| `gemma4:e4b` | ~10 GB | ~6 GB | Summarization, context compression |
| `nomic-embed-text` | ~300 MB | negligible | Embeddings for pgvector |

Thinking mode disabled by default in API calls (`"think": false`) — ~30x speedup for classification, critical for triage latency.

On macOS, run as a LaunchAgent. On Linux, a systemd unit. The reference config binds to `0.0.0.0:11434` so cross-host MCP clients can reach it; bind to `127.0.0.1` if you don't need that.

### llama.cpp (always-on host, port 8081, on-demand)

Optional heavy tier. Hosts a larger MoE model, loaded on demand via `--mmap`.

- Reference model: `unsloth/Qwen3.5-35B-A3B-GGUF` at `UD-IQ3_XXS` quantization (~13 GB on disk).
- MoE architecture: 35B total params, 3B active per token. With `--mmap` the OS pages experts from disk on demand; only shared layers stay resident.
- Use cases: heavy summarization passes, fallback when the cloud model is rate-limited, episodic batch work.

Launched on-demand via a LaunchAgent/systemd unit that the ingester can kickstart; unload after idle.

### Ingester (FastAPI, port 8765)

The busy bee. One Python service that:

1. **Watches local files.** Claude Code JSONLs (`~/.claude/projects/**/*.jsonl`), plus any extra sync-landing directories you configure. The inbox watcher observes `<vault>/_inbox/` for dropped exports.
2. **Receives POSTs** from any agent or shipper that can speak HTTP — `/ingest/document`.
3. **Summarizes** each conversation by calling the summarizer model — output: short markdown summary + frontmatter + cleaned body.
4. **Embeds** each chunk (~500 tokens, 100 overlap) via nomic-embed-text.
5. **Persists** to Postgres (documents + chunks rows) and writes the markdown file to the vault.
6. **Deduplicates** by `(source, source_id)` with content-hash change detection so re-ingestion is idempotent.

Buffers to SQLite if Postgres is unreachable, so you can tolerate storage-host downtime without losing events.

### MCP server (runs on each client)

An MCP stdio server installed wherever you use an MCP-capable client (Claude Code, Claude Desktop, etc.). Exposes:

- `search_brain(query: str, limit: int = 5, sources: list[str] | None = None) -> list[Chunk]`
  The rerank model scores the top-K pgvector hits and returns tight chunks with source metadata, never whole documents.
- `recent_sessions(n: int = 10, source: str | None = None) -> list[DocumentSummary]`
  Chronological recent-first.
- `get_document(vault_path: str) -> str`
  Last-resort full-content retrieval, explicitly requested by the caller.

## Query flow (example)

From Claude Code: *"what did we decide about the Postgres migration?"*

1. Claude Code calls `search_brain("postgres migration decision")` via the local MCP server.
2. MCP server connects to pgvector (directly) and Ollama (for embed + rerank).
3. `nomic-embed-text` embeds the query.
4. pgvector returns top-K chunks by cosine distance.
5. Gemma 4 E2B reranks to top-N by relevance — if the rerank model is already loaded and not contended. If the summarizer is pinned in memory (bulk ingest in progress), the MCP server falls back to raw ANN top-K so search stays fast.
6. MCP returns those N chunks (~2.5K tokens) with metadata.
7. Claude Code answers using that context + its own reasoning.

The primary model never sees the raw vault. The small local models do the librarian work.

## Privacy boundary

- **Stays local:** ingestion, embeddings, summarization, rerank.
- **Leaves local:** only the final curated chunks that get passed to the cloud LLM when the caller explicitly decides to call one. You control this per-session.
- **Never leaves local:** raw JSONL logs, tool-call payloads, vault full-text.

## Failure modes

| Failure | Behavior |
|---|---|
| Postgres host down | Ingester buffers to local SQLite. MCP returns "brain offline". |
| MCP host down | Other client machines' MCP servers continue to work independently. |
| Ollama down | Ingester queues summaries; MCP falls back to raw top-K without rerank. |
| Rerank contention during bulk ingest | MCP's `_rerank_feasible()` detects via `/api/ps` and falls back to ANN top-K. |
| Postgres corrupted | Restore from snapshot; re-derive chunks from vault markdown. |
| Vault sync broken | Your sync layer's own conflict resolution; worst case: restore from snapshot. |
